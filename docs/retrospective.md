# NeuroFlow — Project Retrospective

## Which Task Was Technically Hardest and Why

Task 5 (Retrieval Pipeline) was the most technically demanding. Implementing hybrid retrieval correctly requires understanding three independent retrieval strategies — dense vector search via HNSW, sparse BM25-style search via PostgreSQL full-text search, and metadata-filtered vector search — and then fusing them with Reciprocal Rank Fusion in a way that actually improves results rather than diluting them. The challenge was not the individual components but their interaction: dense retrieval is precise but misses lexical matches; sparse retrieval catches exact terms but cannot generalize; metadata filtering is fast but brittle. Getting RRF to reliably outperform any single strategy required careful tuning of the k parameter and the balance between lists. The cross-encoder reranker added another layer of complexity because it makes one LLM call per candidate, turning a 40-candidate shortlist into 40 parallel API calls — each of which can fail, timeout, or return malformed output. Handling all these failure modes while maintaining end-to-end latency under 4 seconds required the timeout manager and circuit breaker from Task 10 to be in place first.

## Which Design Decision I Would Change With Hindsight

The decision to use an LLM API call for reranking (Task 5) instead of a local cross-encoder model was driven by simplicity, but it was wrong. API-based reranking adds 1–3 seconds of latency per query and costs $0.002–0.01 per reranking pass. A local `cross-encoder/ms-marco-MiniLM-L-6-v2` model runs on CPU in under 200ms for 40 candidates and costs nothing per call. The ADR for chunking strategy correctly anticipated that operational simplicity favors fewer components, but the reranker decision went the other direction without sufficient justification. Given a second pass, I would ship the local cross-encoder from day one and use the API-based approach only as a fallback when the local model is unavailable. The latency improvement alone — 1500ms shaved off every query — would have made the quality improvement sprint in Task 18 much easier.

## What I Learned About Building Production AI Systems

The most important lesson is that AI system quality is not set at deployment — it is continuously measured and managed. Every RAG pipeline is a composition of independent subsystems (ingestion quality, chunk boundaries, embedding model, retrieval strategy, reranking, prompting, and generation), and a failure in any one of them degrades the output in ways that are hard to diagnose without instrumentation at each layer. Distributed tracing (Task 12) revealed that 40% of total query latency was in the reranking step — invisible without spans. Without Prometheus metrics on retrieval latency by strategy, I would not have known which retrieval method was the bottleneck. The lesson is that observability is not optional infrastructure — it is the feedback mechanism that makes improvement possible.

A second lesson is that evaluation must be automated and continuous, not manual and periodic. The RAGAS-style LLM-as-judge (Task 7) runs after every query and catches quality regressions within minutes rather than days. Without automated evaluation, the quality improvement sprint (Task 18) would have required manually reading 30 answers per change — making iteration too slow to be practical.

## What the Quality Improvement Sprint Taught Me

Task 18 reinforced that intuitions about what will improve RAG quality are frequently wrong. I expected query expansion (generating alternative phrasings) to substantially improve retrieval hit rate. In practice, it improved Hit Rate@10 by only 3%, but added 400ms of latency per query and doubled the cost of every retrieval pass. The right response was to keep the feature but make it optional and off by default for latency-sensitive pipelines. Conversely, chunk size reduction (512 → 256 tokens) improved MRR@10 by 12% with no latency cost — a change I had deprioritized because smaller chunks seemed intuitively riskier. The sprint trained a habit of measuring before and after every change rather than relying on intuition, and of treating configuration as a variable rather than a fixed property of the system.
